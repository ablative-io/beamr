## Purpose

Supervision is the highest-value-per-line component. Once beamr provides links, monitors, exit signals, and the trap-exit flag correctly, supervision trees are just Gleam library code — the VM doesn't need to know about supervisors at all. This brief delivers the four primitives that make the system self-healing: bidirectional links (shared fate), unidirectional monitors (observation), exit signals (death notification), and trap_exit (converting death into a message).

## Task

Implement supervision primitives in crates/beamr/src/supervision/. link(A, B) creates a bidirectional bond — each process's link set contains the other. When a process exits, exit signals propagate along links: non-normal reasons kill linked processes (unless they trap exits, in which case the signal becomes a message). monitor(Pid) registers a unidirectional watcher — the monitored process's death sends a DOWN message. The kill signal is untrappable. Normal exits don't propagate. Linking/monitoring an already-dead process delivers the signal immediately.

## Requirements

### R1: Bidirectional links

beamr::supervision::link SHALL create a bidirectional link between two processes. Both processes' link sets SHALL contain the other's PID. link(A, B) and link(B, A) SHALL be equivalent. It SHALL NOT create duplicate links — linking already-linked processes is a no-op.

**Acceptance:**
- link(A, B) adds B to A's links and A to B's links
- After link(A, B), both A.links.contains(B) and B.links.contains(A)
- link(A, B) called twice does not create duplicates
- link(A, A) is a no-op (cannot link to self)

**Files:**
- modify: crates/beamr/src/supervision/link.rs

**Checklist:**

### R2: Exit signal propagation along links

WHEN a process exits THE SYSTEM SHALL send an exit signal with the exit reason to all linked processes. IF the reason is not 'normal' THE SYSTEM SHALL terminate the linked process with the same reason (unless it traps exits). IF the reason is 'normal' THE SYSTEM SHALL NOT terminate linked processes. It SHALL NOT skip any linked process — all links are visited.

**Acceptance:**
- Process A exits with reason 'crash'; linked process B receives the exit signal and exits with 'crash'
- Process A exits with reason 'normal'; linked process B is NOT terminated
- Process A linked to B and C: both B and C receive exit signals when A dies
- Exit signals propagate transitively: A→B→C, A dies, B dies, C dies

**Files:**

### R3: Trap exit flag

WHEN a process has trap_exit=true THE SYSTEM SHALL convert incoming exit signals into {'EXIT', Pid, Reason} messages delivered to the process's mailbox instead of terminating it. The process receives the message and can handle it (this is how supervisors work). It SHALL NOT trap the 'kill' signal — kill is always fatal.

**Acceptance:**
- Process B with trap_exit=true receives {'EXIT', A_pid, crash} as a mailbox message when A dies
- Process B with trap_exit=false exits when A dies with a non-normal reason
- Process B with trap_exit=true still dies when sent a 'kill' signal

### R4: Kill signal — untrappable termination

WHEN a process receives an exit signal with reason 'kill' THE SYSTEM SHALL terminate it unconditionally, even if trap_exit is true. The terminated process SHALL propagate exit reason 'killed' (not 'kill') to its own links. It SHALL NOT allow any process to survive a kill signal.

**Acceptance:**
- A process with trap_exit=true receiving kill signal is terminated
- The killed process propagates reason 'killed' (not 'kill') to its links
- Links of the killed process that trap exits receive {'EXIT', Pid, killed}

**Files:**
- modify: crates/beamr/src/supervision/link.rs

**Checklist:**

### R5: Monitors — unidirectional observation

beamr::supervision::monitor SHALL register a unidirectional monitor. monitor(Watcher, Target) returns a unique Reference. WHEN the target exits THE SYSTEM SHALL send a {'DOWN', Ref, process, Pid, Reason} message to the watcher. The watcher is NOT terminated — monitors are observation-only. demonitor(Ref) SHALL remove the monitor; no DOWN message is sent after demonitor.

**Acceptance:**
- monitor(A, B) returns a unique Reference
- When B exits, A receives {'DOWN', Ref, process, B_pid, Reason} in its mailbox
- A is NOT terminated when B exits (monitors are non-fatal)
- demonitor(Ref) prevents the DOWN message from being sent

### R6: Unlink and already-dead edge cases

unlink(A, B) SHALL remove the bidirectional link — no exit signal after unlink. Monitoring an already-exited process SHALL immediately send a DOWN message. Linking to an already-exited process SHALL immediately send an exit signal. It SHALL NOT silently swallow these edge cases — the caller must observe the death.

**Acceptance:**
- unlink(A, B) removes B from A's links and A from B's links
- After unlink, A's death does not affect B
- monitor(A, dead_process) immediately delivers {'DOWN', Ref, process, Pid, Reason}
- link(A, dead_process) immediately delivers an exit signal to A

### R7: Normal exit propagation for process lifecycle

WHEN a process exits with reason 'normal' THE SYSTEM SHALL deallocate its heap and remove it from the process table (completing the C55 split from B-010). Normal exits SHALL propagate along links but SHALL NOT cause linked processes to exit. It SHALL NOT leak heap memory or leave stale entries in the scheduler.

**Acceptance:**
- A normally-exiting process's heap is deallocated
- A normally-exiting process is removed from the process table
- Linked processes are not terminated by a normal exit
- No stale PID entries remain in the scheduler's wait set or run queues
