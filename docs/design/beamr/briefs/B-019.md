# B-019: Implement scheduler priority queues and dirty scheduler pool

> **Cluster:** beamr
> **Depends on:** B-012, B-008

## Purpose

The core scheduler (B-012) delivers the basic thread pool, run queues, and work stealing. This brief adds the two remaining scheduler features: priority queues (high and max priority processes are scheduled before normal) and the dirty scheduler pool (a separate thread pool for long-running native calls that would otherwise block normal scheduler threads and break the fairness promise).

## Task

Extend the scheduler in crates/beamr/src/scheduler/ with priority queues and the dirty scheduler pool. Each worker thread gets three queues: max, high, and normal priority — dequeuing checks max first, then high, then normal. The dirty pool is a separate set of OS threads that handle native calls marked as dirty (via B-008's NativeEntry.is_dirty flag). When the interpreter hits a dirty native call, the process migrates to the dirty pool for the duration; the normal worker thread is freed immediately.

## Requirements

### R1: Priority queue support

Each scheduler worker SHALL maintain three queues: max priority, high priority, and normal priority. WHEN selecting the next process to run THE SYSTEM SHALL dequeue from max first, then high, then normal. Process priority SHALL be settable at spawn time and changeable at runtime. It SHALL NOT starve normal-priority processes indefinitely — if max/high queues are always occupied, normal processes must still make progress (fairness within a bounded window).

**Acceptance:**
- A max-priority process is dequeued before a high-priority process
- A high-priority process is dequeued before a normal-priority process
- With only normal-priority processes, scheduling behaves identically to B-012
- A normal-priority process is not starved if high-priority processes are present (runs within 10 scheduling rounds)

### R2: Dirty scheduler thread pool

beamr::scheduler::dirty SHALL define a DirtyPool — a separate pool of OS threads dedicated to long-running native calls. WHEN the interpreter encounters a native call with is_dirty=true THE SYSTEM SHALL migrate the calling process to the dirty pool, freeing the normal worker thread immediately. WHEN the dirty call completes THE SYSTEM SHALL return the process to a normal run queue. It SHALL NOT run dirty calls on normal scheduler threads.

**Acceptance:**
- A dirty native call executes on a dirty pool thread, not a normal scheduler thread
- The normal worker thread is freed immediately when a process enters the dirty pool
- After the dirty call completes, the process is placed back on a normal run queue
- The dirty pool has a configurable number of threads (default: number of CPU cores)

### R3: Dirty dispatch integration

WHEN the interpreter resolves a native call THE SYSTEM SHALL check NativeEntry.is_dirty (from B-008). IF dirty THE SYSTEM SHALL yield the process with a DirtyCall status, and the scheduler SHALL dispatch it to the dirty pool. After completion, the process resumes on a normal thread with the native call's result in the appropriate register. It SHALL NOT block the calling process on the normal thread while waiting for the dirty call.

**Acceptance:**
- An interpreter call to a dirty-flagged BIF yields the process with DirtyCall status
- The scheduler picks up DirtyCall processes and sends them to the dirty pool
- After dirty completion, the result is in the expected register and the process resumes normally
- Normal scheduler threads are never blocked by dirty calls
