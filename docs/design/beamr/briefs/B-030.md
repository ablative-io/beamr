## Purpose

gleam_erlang's process module wraps BEAM's selective receive with a typed selector abstraction. A `Selector` is a list of handler functions — `select/1` scans the mailbox and applies the first matching handler. This is the foundation of gleam_otp's actor message loop. Without it, no Gleam OTP code can receive messages. Implement as native Rust BIFs registered under `gleam_erlang_ffi` MFAs.

## Task

Implement the selector system in `crates/beamr/src/native/selector_ffi.rs`. A selector is a term (list of handler tuples). `select` scans the process mailbox, applies each handler in order, and returns the first successful result. Register all functions under the `gleam_erlang_ffi` module atom.

## Boundaries

- This brief SHALL NOT implement gleam_erlang_ffi functions unrelated to selectors (those are in B-031)
- This brief SHALL NOT implement ETS or persistent_term operations
- This brief SHALL NOT modify the mailbox implementation — use existing selective receive primitives

## Requirements

### R1: new_selector/0 and basic selector structure

`gleam_erlang_ffi:new_selector/0` SHALL return an empty selector (NIL — an empty list). A selector is a list of handler entries. Each entry is a tuple `{Tag, Handler}` where Tag is an atom or term used for matching and Handler is a fun/1 that transforms the matched message.

**Acceptance:**
- new_selector() returns an empty list (NIL)
- The selector representation is a standard BEAM list of tuples

**Files:**
- create: crates/beamr/src/native/selector_ffi.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C150

### R2: insert_selector_handler/3, map_selector/2, merge_selector/2, remove_selector_handler/2

`insert_selector_handler/3` SHALL accept (Selector, Tag, Handler) and prepend `{Tag, Handler}` to the selector list. `map_selector/2` SHALL accept (Selector, MapFun) and wrap each handler to apply MapFun to its result. `merge_selector/2` SHALL concatenate two selectors. `remove_selector_handler/2` SHALL remove entries with a matching tag.

These operations build and compose selectors before `select` is called.

**Acceptance:**
- insert_selector_handler adds a handler to the selector
- map_selector transforms handler outputs
- merge_selector combines two selectors
- remove_selector_handler removes by tag
- Operations compose correctly (insert then merge then select)

**Files:**
- modify: crates/beamr/src/native/selector_ffi.rs

**Checklist:**
- C150

### R3: select/1 and select/2

`select/1` SHALL scan the calling process's mailbox, testing each message against the selector's handlers in order. The first handler whose tag matches the message consumes the message and returns `Handler(Message)`. If no message matches any handler, the process suspends (Waiting) until a new message arrives.

`select/2` SHALL behave like `select/1` but accept a timeout in milliseconds. If no matching message arrives within the timeout, return `{error, nil}`.

These functions need access to the process mailbox. Since BIFs receive ProcessContext (not the full Process), add a `SelectFacility` trait that the scheduler implements, providing mailbox scanning and process suspension.

**Acceptance:**
- select(Selector) with matching message returns handler result
- select(Selector) with no matching message suspends the process
- select(Selector, Timeout) returns {error, nil} on timeout
- Messages not matching any handler remain in the mailbox
- The first matching handler wins (order matters)
