# B-042: Process version pinning and safe version transitions

> **Cluster:** beamr
> **Depends on:** B-041
> **Checklist:**
> - C110 — Module registry maps module names to loaded Module structs
> - C130 — Process struct holds x-registers, y-registers, stack, mailbox, code position

## Purpose

When a module is hot-loaded, running processes must continue executing the version they started with until they make a fully-qualified external call (BEAM semantics). Currently, the interpreter re-resolves the module from the registry on every instruction cycle, meaning a process that yields mid-function will silently switch to the new version on resume — with an instruction pointer that indexes into the old code layout. This causes undefined behaviour: wrong instruction execution, out-of-bounds panics, or silent corruption.

Process version pinning ensures a process is bound to a specific module version and only transitions to the new version at well-defined safe points.

## Boundaries

- This brief SHALL NOT implement closure version pinning (that's B-044)
- This brief SHALL NOT implement import re-resolution (that's B-043)
- This brief SHALL handle stack frame return addresses alongside code position pinning
- This brief SHALL ensure that intra-module calls (call, call_only, call_last) stay on the pinned version

## Requirements

### R1: Per-stack-frame module version pinning

Each stack frame must hold a reference to the specific module version it was executing when the frame was pushed. A single global `pinned_module` on Process is insufficient — a process calling A → B → C has three modules on its call stack, each potentially at different versions. Each frame pins its own version via `Arc<Module>`.

**Acceptance:**
- `StackFrame` gains a `pinned_module: Arc<Module>` field alongside existing `return_module: Atom` and `return_ip: usize`
- `push_frame` takes an additional `module: Arc<Module>` parameter — the module version the caller is executing
- `pop_frame` returns a `ReturnPoint` that includes the `Arc<Module>` so the interpreter can resume in the correct version
- `ReturnPoint` gains a `module_version: Arc<Module>` field
- The `Arc` keeps the old module version alive even after it's replaced in the registry or purged
- Test: push frames for modules A v1, B v1, C v1. Reload A to v2 and B to v2. Pop C's frame — returns to B v1. Pop B's frame — returns to A v1 (not v2). Both old versions stayed alive via Arc.

**Files:**
- modify: crates/beamr/src/process/stack.rs (StackFrame, ReturnPoint, push_frame, pop_frame)

**Checklist:** C130

### R2: Process current-module pin

Add a field to Process that tracks the module version currently being executed. This is the "active" pin — it's updated on module transitions and used by the interpreter to avoid per-instruction registry lookups.

**Acceptance:**
- Process gains a `current_module: Option<Arc<Module>>` field
- When a process starts executing a function in module X, `current_module` is set to the resolved version
- `current_module` is set at process spawn, at fully-qualified external call entry, and on return from external calls (restored from stack frame)
- `current_module` is cleared on process exit
- Accessor: `Process::current_module() -> Option<&Arc<Module>>`
- Mutator: `Process::set_current_module(module: Arc<Module>)`

**Files:**
- modify: crates/beamr/src/process/mod.rs

**Checklist:** C130

### R3: Interpreter uses pinned module

Modify the interpreter run loop to use the process's current_module instead of re-resolving from the registry on every iteration. The registry lookup is only used when the process needs to transition to a different module.

**Acceptance:**
- `run_loop` checks `process.current_module()` before falling back to `registry.lookup()`
- If `process.current_module()` matches the current `code_position.module`, use the pinned version directly — no registry lookup
- If `code_position.module` differs from the current module's name (cross-module jump happened), resolve from registry and update the pin
- Intra-module calls (`call`, `call_only`, `call_last`) never trigger a registry re-lookup — they use the pinned version
- Test: load module A v1 with function `foo` that calls `bar` internally. Start process running `foo`. Mid-execution (between `foo` and `bar`), load module A v2. Process continues executing v1 for both `foo` and `bar`.

**Files:**
- modify: crates/beamr/src/interpreter/mod.rs (run_loop)

**Checklist:** C130

### R4: External calls trigger version transition — outgoing only

Fully-qualified external calls (`call_ext`, `call_ext_only`, `call_ext_last`) are the ONLY safe points where a process transitions to a new version. Version transitions happen on outgoing calls, NEVER on returns.

BEAM semantics: when process in A v1 calls B, the return address is saved in A v1's code. When B returns, the process resumes in A v1 — not the current version of A. The process stays on A v1 until it makes another fully-qualified outgoing call (the classic BEAM upgrade pattern: `?MODULE:loop(State)` is a self-referencing fully-qualified call that picks up the current version).

**Acceptance:**
- When `call_external_target` resolves a `ResolvedImportTarget::Code` target, it saves the current module version in the stack frame (R1), then looks up the **current** version of the target from the registry
- The process's `current_module` is updated to the target module's current version
- On return from an external call, the process restores `current_module` from the stack frame's `pinned_module` — resuming in the SAME version it was executing before the call, NOT the current version from the registry
- The classic upgrade path works: if a process calls `?MODULE:function()` (fully-qualified self-call), it picks up the current version. If it calls `function()` (local call), it stays on its current version.
- Test: process in module A v1 calls module B. While in B, module A is reloaded to v2. When B returns, the process resumes in A v1 (not v2). The process then calls A:loop/0 (fully-qualified self-call) — NOW it picks up A v2.

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (call_external_target, return_)

**Checklist:** C110, C130

### R5: Scheduler slice entry respects pinning

The scheduler's `execute_slice` currently resolves the module at slice entry and passes it as `initial_module`. It should check the process's current_module pin first.

**Acceptance:**
- `execute_slice` checks `process.current_module()` before doing a registry lookup
- If the process has a current module matching `code_position.module`, pass the pinned module to the interpreter
- If no pin exists (fresh process), resolve from registry and set the pin
- A process that yields mid-execution and is re-scheduled continues with its pinned version, not whatever is currently in the registry
- Test: start a process with a large reduction budget. Mid-execution, reload the module. The process continues on the pinned version until it yields and re-enters execute_slice. Verify the IP is still valid for the pinned version.

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (execute_slice)

**Checklist:** C110, C130
